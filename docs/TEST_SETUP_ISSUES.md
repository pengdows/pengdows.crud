# Test Setup Issues

This document tracks identified issues with the `pengdows.crud` test suite and the usage of `fakeDb`.

## Current Pain Points

### 1. Brittle Manual Mocking (`SetIdPopulationResult`)
The `SetIdPopulationResult` helper (and similar extensions in `FakeDbFactoryTestExtensions.cs`) manually seeds a fixed list of 7-8 connections. This is intended to survive the `DatabaseContext` initialization and subsequent operations.
- **Risk:** If the initialization sequence changes (e.g., an extra version check or a new session setting), tests break because they run out of queued connections.
- **The "Lost" Connection:** In `Standard` mode, the context opens and immediately disposes an initial connection for detection. Manual mocking must "waste" one connection to account for this.

### 2. ~~Lack of Identity Capture in `FakeDataStore`~~ — RESOLVED

`FakeDataStore` now handles all identity-return clauses:
- `INSERT ... RETURNING id` (PostgreSQL) — handled in `ExecuteReader`
- `SCOPE_IDENTITY()`, `LAST_INSERT_ID()`, `LAST_INSERT_ROWID()`, `LASTVAL()`, `@@IDENTITY` — all return `_lastInsertId`
- Auto-increments `_lastInsertId` on every INSERT

Tests with `EnableDataPersistence = true` no longer need manual ID enqueuing for standard create flows.

### 3. Verification Gaps (The "Batch" Problem)
Many tests (notably `TableGatewayBatchTests.cs`) set `EnableDataPersistence = true` but only verify that the return value is non-negative.
- **Missing Check:** They do not perform a follow-up `RetrieveAsync` to prove that the data was actually persisted correctly into the right columns.

### 4. Mixed Test Environments
The suite is split between `RealSqliteContextTestBase` (using a real in-memory SQLite DB) and `fakeDb`. 
- **Inconsistency:** Some logic is tested against a real DB (slow, SQLite-only syntax), while other logic uses a fake DB (fast, but "dumb" SQL parsing).

---

## Recommended Strategy

### 1. Protocol/Stateful Mocking
Move away from "Blind Sequence Mocking" (fixed lists of connections) toward **Command-Aware Mocking**.
- Leverage `ScalarResultsByCommand` for version queries and session settings.
- Use a **Persistent Handler** approach where the factory can serve an infinite number of "initialization-style" queries without consuming the operation results.

### 2. Standardize on `SingleConnection` for Unit Tests
Unless testing the connection strategy itself, unit tests should prefer `DbMode.SingleConnection`. 
- **Benefit:** It eliminates the "Standard mode connection waste" and makes the connection lifecycle deterministic and easy to mock.

### 3. Enhance `FakeDataStore` for Identity Capture
Update the `FakeDataStore` parser to recognize and handle identity-return clauses.
- **Goal:** `CreateAsync` should just work with persistence enabled, without manual ID enqueuing.

### 4. Round-Trip Verification
Update high-impact tests (Batch, Upsert, Create) to always perform a `Retrieve` after a write. 
- **Why:** This verifies that the SQL generated by the gateway is actually valid and that values are mapped to the correct columns in both directions.
